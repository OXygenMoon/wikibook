**功能设计**

用户希望在徽章管理中新增两种获取条件：

1. **全员获取 (All Users)**：

   * **老用户**：在保存徽章设置时，立即为所有现有用户发放。

   * **新用户**：在用户注册（`/register`）成功后，自动发放。

   * **实现**：新增条件类型 `all_users`。

2. **限时登录天数 (Login Days in Range)**：

   * **设置**：需要设定 `start_time` (开始日期)、`end_time` (结束日期) 和 `condition_value` (累计天数)。

   * **判定**：在该时间段内，用户有过活跃记录（通过 `StudySession` 判断，因为 `StudySession` 记录了用户的具体活动时间段，比单纯的 `UserActiveStatus` 更适合做历史回溯）。

   * **实现**：

     * 新增条件类型 `login_days_in_range`。

     * `Badge` 模型需要新增 `start_time` 和 `end_time` 字段。

     * 在 `check_and_award_badges` 中增加逻辑：查询 `StudySession` 在指定时间范围内的去重日期数。

**数据库变更**

* `Badge` 表新增：

  * `start_time` (Date, nullable)

  * `end_time` (Date, nullable)

**代码修改点**

1. **`app.py`** **模型与路由**：

   * 更新 `Badge` 模型，添加新字段。

   * 更新 `manage_badges` 和 `edit_badge` 路由，处理新字段的保存。

   * 更新 `register` 路由，添加“全员获取”徽章的发放逻辑。

   * 更新 `check_and_award_badges` 函数：

2. **前端模板**：

   * 更新 `templates/admin/manage_badges.html` 和 `edit_badge.html`。

**迁移策略**

由于涉及数据库 Schema 变更（SQLite），如果不使用 Flask-Migrate，直接修改 Model 会导致错误。简单的做法是：

* 用户本地开发环境：可以直接删除 db 文件重建（但这会丢失数据）。

* 或者，在应用启动时检查列是否存在，不存在则 ALTER TABLE。鉴于这是开发辅助，我将使用 `sqlite3` 命令行或在 `app.py` 启动前执行一次性 SQL 来添加列。

* **决定**：在 `app.py` 中添加一个 `upgrade_db()` 函数，利用 `sqlite3` 或 `sqlalchemy` 执行 `ALTER TABLE badge ADD COLUMN start_time DATE;` 等语句，确保平滑升级。
