# Interactive Heartbeat & Online Status Plan

I will refactor the heartbeat mechanism to be "User Activity Driven" rather than "Timer Driven". This ensures that users who leave the tab open but go idle (e.g., computer sleep, stepping away) are correctly marked as offline.

## 1. Frontend Logic (`templates/base.html`)
-   **Remove the fixed Interval**: Delete `setInterval(sendHeartbeat, 30000)`.
-   **Implement Activity Listeners**:
    -   Listen for user interaction events: `click`, `keydown`, `scroll`, `mousemove`.
    -   **Throttling**: Ensure the heartbeat is sent at most **once every 30 seconds** (or 1 minute), but *only* if an interaction occurred.
    -   **Immediate Trigger**: The first interaction after the throttle period expires will trigger an immediate heartbeat.
-   **Visibility API**:
    -   Listen for `visibilitychange` event.
    -   Stop sending heartbeats if `document.hidden` is true (tab backgrounded).
    -   Send an immediate heartbeat when `document.hidden` becomes false (user returns).

## 2. Backend Logic (`app.py`)
-   **No major changes needed** to the `/api/heartbeat` endpoint logic itself.
-   The existing "5-minute timeout" logic (`User.is_online`) will naturally mark users as offline if the frontend stops sending heartbeats due to inactivity.

## 3. Implementation Steps
1.  **Refactor JS in `base.html`**:
    -   Create a `lastHeartbeatTime` variable.
    -   Create an `attemptHeartbeat()` function that checks `Date.now() - lastHeartbeatTime > 30000`.
    -   Bind `attemptHeartbeat` to `window` events (`click`, `scroll`, etc.).
    -   Ensure one initial heartbeat is sent on load.

This approach guarantees that "Online" means "Actually interacting with the page", solving the issue of idle/sleeping users being shown as online.
