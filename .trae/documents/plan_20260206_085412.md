# 系统通知功能开发计划

## 1. 数据库模型设计

我们需要新建两个模型来存储通知内容和用户的确认状态。

### 新增模型

* **SystemAnnouncement (系统通知)**

  * `id`: 主键

  * `title`: 标题

  * `content`: 内容 (Markdown格式，支持富文本和图片)

  * `created_at`: 创建时间

  * `updated_at`: 更新时间

  * `created_by_id`: 创建者 (管理员)

  * `is_active`: 是否启用 (方便软删除或草稿)

* **UserAnnouncementConfirmation (用户确认记录)**

  * `id`: 主键

  * `user_id`: 用户ID

  * `announcement_id`: 通知ID

  * `confirmed_at`: 确认时间

## 2. 后端接口开发 (app.py)

### 管理员接口

* `GET /admin/announcements`: 通知列表页，显示所有通知及其状态。

* `GET/POST /admin/announcements/new`: 创建新通知，使用 EasyMDE 编辑器（支持图片上传）。

* `GET/POST /admin/announcements/<id>/edit`: 编辑现有通知。

* `GET /admin/announcements/<id>/stats`: 查看通知的确认统计（哪些用户已确认，哪些未确认）。

* `POST /admin/announcements/<id>/delete`: 删除/停用通知。

### 用户/API 接口

* `GET /api/announcements/check`: 检查当前用户是否有未确认的通知。

  * 返回：如果有未确认，返回最新的一条未确认通知详情；如果都确认了但有历史通知，返回显示“历史入口”的标志。

* `POST /api/announcements/<id>/confirm`: 用户点击确认收到。

* `GET /api/announcements/history`: 获取当前用户可见的历史通知列表。

## 3. 前端界面开发

### 管理员界面 (templates/admin/)

* `manage_announcements.html`: 表格展示通知列表，提供增删改查入口。

* `edit_announcement.html`: 表单页，集成 EasyMDE 编辑器，配置图片上传接口（复用现有的 `/upload`）。

* `announcement_stats.html`: 展示已读/未读用户列表。

### 用户界面 (templates/base.html & JS)

* **全局通知检测**: 在 `base.html` 中添加 JavaScript，页面加载时调用 `/api/announcements/check`。

* **弹窗显示**:

  * 如果有未确认通知，使用 DaisyUI Modal 弹窗显示通知内容。

  * 弹窗底部包含“确认收到”按钮，点击后调用确认接口并关闭弹窗。

* **Logo 改造**:

  * 修改 Navbar 中的 "WikiBook" 文字结构，将 `i` 字母包裹在相对定位容器中。

  * 当用户确认通知后（或检测到有历史通知时），在 `i` 上方显示一个小圆点（红色或醒目颜色）。

  * 点击圆点弹出“历史公告”模态框，展示过往通知列表。

## 4. 实施步骤

1. **数据库变更**: 定义模型并执行数据库迁移（`db.create_all()` 自动处理）。
2. **后端逻辑**: 实现上述 Admin 和 API 路由。
3. **前端 Admin**: 完成管理页面的 HTML 模板。
4. **前端 User**:

   * 修改 `base.html` 结构适配 Logo。

   * 编写 JS 逻辑处理弹窗、确认交互和红点显示。

   * 集成 Markdown 渲染（复用 `mistune` 或前端渲染）用于展示通知内容。

## 5. 验证计划

* **管理员**: 测试发布包含图片的富文本通知，修改通知，查看统计数据是否准确。

* **新用户/未读用户**: 登录后应立即看到弹窗，点击确认后弹窗消失，Logo 出现红点。

* **已读用户**: 登录后不弹窗，但 Logo 上有红点，点击可查看历史。

* **无通知状态**: 系统无通知时，Logo 无红点，不弹窗。
