**问题确认**

1.  **图片裂开原因**：数据库中存储的某些徽章图标路径（如 `uploads/badges/...`）包含了 `uploads/` 前缀。现有的 `badge_icon_url` 过滤器无条件地再次添加 `uploads/` 前缀，导致生成的 URL 为 `/static/uploads/uploads/badges/...`，路径错误从而无法加载图片。
2.  **贴纸入口/显示问题**：
    *   **显示为文件名**：Markdown 编辑器（EasyMDE）上传图片时，因后端接口返回格式不符合编辑器预期（缺 `url` 字段），导致插入的是文件名文本而非图片 Markdown 语法。
    *   **入口不见了**：极可能是指因为图片加载失败（裂开），导致原本应该显示图片（贴纸/徽章）的地方显示异常，或者是指 Markdown 编辑器里的图片插入体验受阻。

**修复计划**

1.  **修复路径解析逻辑 (app.py)**：
    *   修改 `badge_icon_url` 过滤器：在拼接路径前检查字符串是否已包含 `uploads/` 前缀。若已包含，则不再重复添加。
2.  **修复图片上传接口 (app.py)**：
    *   修改 `upload_file` 路由：将返回 JSON 格式调整为 EasyMDE 默认支持的 `{"success": 1, "url": "..."}`。这能同时解决“贴纸显示为文件名”的问题。
3.  **适配前端上传逻辑 (templates/admin/edit_badge.html)**：
    *   更新 `uploadBadgeIcon` 函数：适配新的接口返回格式，从 `data.url` 获取图片路径。
4.  **增强 Badge 模型定义 (app.py)**：
    *   将 `Badge.icon` 字段长度从 50 增加到 255，以防止长路径被截断（虽然 SQLite 宽容，但为了严谨性）。

**验证步骤**

1.  重启应用。
2.  访问首页/在线用户列表，检查之前的“裂图”是否恢复正常显示。
3.  进入徽章编辑页，尝试上传新图片，确认能自动填入路径且路径正确。
4.  进入 Wiki/笔记编辑页，使用工具栏图片按钮上传图片，确认能正确插入 `![](/static/uploads/...)` 格式并预览显示图片。
